00 서비스 구성 — 뭘 새로 만들고 뭘 재활용하는가

채팅 파이프라인 인프라(라우팅·연결·메시지·팬아웃)는 이미 모듈로 존재한다. 새 도메인을 위한 신규 모듈은 셋 — 매칭 코어(matching-service) · 사람 추천(recommendation-service) · 오픈채팅(openchat-service)이고, 각각 부하·장애 프로파일이 다르다(hot key / 랭킹 서빙 / 방 fanout)라서 분리한다. 나머지는 전부 기존 모듈 확장 또는 재활용 — "새 도메인을 최소 신규 코드로 얹는다"가 이 설계의 뼈대.
모듈구분역할Phase
matching-service 🆕 신규 좋아요/패스 수집 → like-relay 파티션 직렬화 → 멱등 매칭 성사 → outbox (프로젝트의 심장) P1 🔥
recommendation-service 🆕 신규 사람 추천 피드 — geo 반경·필터(나이/거리/관심사)·랭킹·본 사람 제외(bloom). 랭킹 서빙 read-heavy P2
openchat-service 🆕 신규 오픈채팅방 탐색/검색/입장. 방 메타 관리 + 대규모 방 fanout·presence heavy P3
user-service ♻️ 확장 프로필·위치·관심사·선호 필드 추가 (users 테이블 확장) P0
message-service ♻️ 확장 channel.type(DIRECT|OPEN) 한 컬럼 추가로 1:1·오픈방 통일 P1 · P3
fanout-delivery-service ♻️ 확장 match-fanout 구독 추가(매칭 성사 → 양쪽 푸시 + 방 자동 오픈) · 대규모 오픈방 fanout·presence P1 · P3
connection-service ♻️ 재활용 인스턴스 라우팅(connection-instance-{id}) — 수정 없이 그대로 사용 —
api-gateway · eureka-server ♻️ 재활용 게이트웨이 · 서비스 디스커버리 — 라우트만 추가 —

인프라 (모듈 아님)

Redis — geo:users(GEO 반경검색) · feed:{userId}(추천 큐) · seen:{userId}(Bloom, 본 사람 제외)
Debezium (CDC) — outbox binlog → match-fanout 무손실 배달
Kafka — like-relay · match-fanout · 기존 message/read 토픽

01 데이터 모델

user-service ♻️ 확장

users 테이블에 프로필 필드 추가

users:  id, email, name, gender, birth_year, bio,
        interests(JSON tags), lat, lng, geohash,
        photos(JSON url list),          -- placeholder URL이면 충분
        pref_gender, pref_age_min/max, pref_distance_km

matching-service 🆕

likes:   from_user_id, to_user_id (복합 PK),
         type(LIKE|PASS|SUPER), created_at
         -- 중복 좋아요는 PK로 멱등 흡수

matches: match_id PK,
         user_lo_id, user_hi_id,        -- 항상 정렬 저장 (lo < hi)
         channel_id, status(ACTIVE|UNMATCHED), created_at,
         UNIQUE(user_lo_id, user_hi_id) -- ★ 정확히 한 번 매칭 ★

outbox:  (기존 패턴 그대로) MATCH_CREATED 이벤트

message-service ♻️

channel.type 컬럼만 추가 (DIRECT | OPEN). 1:1 매칭방도 오픈방도 같은 channel 테이블로 통일.

recommendation-service 🆕

사람 추천 피드. 상태를 거의 Redis에 두고 users(user-service)를 조합해 서빙 — 자체 RDB 테이블은 사실상 없음.

-- 상태는 아래 Redis 자원 + user-service.users 조합으로 서빙
-- (필요 시) reco_log: user_id, shown_user_id, shown_at  -- 노출/AB 분석용(옵션)

openchat-service 🆕

rooms:        room_id PK, title, category, member_count, created_at
room_members: room_id, user_id (복합 PK), joined_at

방 메타/멤버십만 소유. 실제 대화는 message-service(channel.type=OPEN), 대량 전송은 fanout-delivery-service 재활용.

Redis recommendation-service 자원

geo:users        (GEO)        -- GEOADD userId lng lat → 반경 검색
feed:{userId}    (List)       -- 추천 후보 큐
seen:{userId}    (Bloom/Set)  -- 이미 본 사람 제외

02 토픽 토폴로지

토픽파티션 키Pub → Sub용도
like-relay pair(a,b) client → matching-service 좋아요/패스 수집 · 직렬화
match-fanout matchId Debezium CDC → fanout-delivery-service 매칭 성사 → 양쪽 알림 + 방 생성
message-relay / message-fanout channelId (기존 ♻️) 1:1 · 오픈방 채팅
read-relay / read-fanout channelId (기존 ♻️) 읽음 처리
connection-instance-{id} — (기존 ♻️ connection-service) 인스턴스 라우팅
핵심: like-relay의 파티션 키를 두 사람 쌍의 해시 pair(a,b) = hash(min, max)로 잡으면, A→B와 B→A 좋아요가 같은 파티션 → 같은 컨슈머 스레드에 순서대로 들어옵니다. 채팅에서 channelId로 메시지 순서를 보장하던 그 메커니즘이, 여기선 분산 락 없이 매칭을 직렬 판정하는 장치가 됩니다.

03 매칭 성사 플로우 — 메인 클라이맥스

  1. A가 B에게 좋아요 → like-relay [key=pair(A,B)]
  2. matching consumer: like 저장(PK 멱등). 역방향(B→A) 있나? → 없음, 종료
  3. (나중에) B가 A에게 좋아요 → like-relay 같은 파티션 → 같은 컨슈머
  4. 역방향(A→B) 있나? → 있음!
    → matches INSERT (UNIQUE(lo,hi)로 중복 매칭 차단)
    → outbox(MATCH_CREATED, recipients=[A,B], new channel_id) ※ 한 트랜잭션
  5. Debezium CDC → match-fanout → fanout-delivery-service
    → Redis ws:user 조회 → 양쪽 connection-instance로 푸시
    → "매칭 성사!" 알림 + DIRECT 채팅방 자동 오픈
  6. 두 사람 1:1 채팅 시작 (기존 채팅 파이프라인 그대로)
단일 컨슈머 직렬화 + UNIQUE 제약의 이중 안전장치라, 양쪽이 같은 ms에 좋아요를 눌러도 채팅방은 정확히 하나. 이게 부하 테스트에서 "동시 스와이프 폭주에도 중복 매칭 0" 그래프로 증명됩니다.

04 구현 로드맵

Phase 0
기반 · 시드
user-service 프로필/위치/관심사/선호 확장 + 합성 프로필 수만 개 생성기 (테스트 토대)
Phase 1 🔥
매칭 코어 — 데모의 심장
좋아요/패스 API → like-relay 직렬화 → 멱등 매칭 성사 → match-fanout → 실시간 알림 + DIRECT 채팅방 자동 생성 → 1:1 채팅 연결.
Phase 2
추천 피드 · recommendation-service
geo 반경 + 필터(나이/거리/관심사) + 랭킹 + 본 사람 제외(bloom) + Redis 캐시 + 커서 페이지네이션
Phase 3
오픈채팅 · openchat-service + message/fanout 재활용
방 탐색/검색/입장, 대규모 방 fanout, presence(접속자 수). 방 메타는 openchat-service가 소유하고 대화·전송은 기존 채팅 파이프라인을 그대로 재활용한다.
Phase 4
부하 · 정합성 증명
k6 스와이프 폭주 → 중복 매칭 0 / 인기 유저 hot key / 추천 서빙 부하 + Grafana 대시보드
옵션
moderation
차단 · 신고 · 필터링
Phase 1이 기존 자산(멱등성 · Outbox · CDC · fanout · 채팅)을 가장 많이 재활용하면서 새 도메인의 핵심을 보여줘서, 여기부터 손대는 걸 추천합니다.

05 왜 좋아요를 DB에 바로 안 쓰고 Kafka를 거치는가

면접에서 거의 100% 나오는 질문 — 이 프로젝트 백엔드 깊이 그 자체.

"그냥 DB에 바로 쓰면" 뭐가 문제인가

순진한 로직: ① like(A→B) 저장 → ② 역방향 like(B→A) SELECT → ③ 있으면 match 생성. 인스턴스가 N개라 A의 요청은 인스턴스1, B의 요청은 인스턴스2가 동시에 처리할 수 있다.

시각   인스턴스1 (A→B 처리)            인스턴스2 (B→A 처리)
──────────────────────────────────────────────────────────────
 t1    like(A→B) 저장
 t2                                    like(B→A) 저장
 t3    역방향(B→A) 있나? SELECT
        → 아직 commit 전이라 "없음"
 t4                                    역방향(A→B) 있나? SELECT
                                        → "없음"
 t5    match 안 만듦                    match 안 만듦
──────────────────────────────────────────────────────────────
결과: 둘 다 서로 좋아하는데 매칭이 안 됨!  ❌ (유실)
중복 매칭(두 개 생김)은 차라리 UNIQUE(lo,hi)로 막을 수 있는데, 위처럼 둘 다 "상대 거 아직 안 보임" 상태로 엇갈리면 매칭이 아예 누락된다. 두 사람 다 "왜 매칭 안 되지?" 하고 이탈 — 전형적인 read-then-write race condition.

해결법들과 트레이드오프

방법설명문제
A — DB 락
(SELECT FOR UPDATE)		 쌍(pair)마다 비관적 락 → 한 번에 한 요청만 인기 유저에 락 경합 폭발 · 데드락 위험 · 잠글 "기준 행"이 애매 · 대용량에서 안 좋음
B — 분산 락
(Redis Lock)		 외부 락으로 직렬화 인프라 추가 + 락 획득/해제/만료 관리 복잡
C — Kafka 파티션 직렬화 ★
(우리 선택)		 파티션 키 = pair(a,b) 락 없이 쌍별 단일 작성자(single-writer per pair) 획득

방법 C — Kafka가 빛나는 지점

A→B 좋아요  ─┐
              ├─ pair(A,B) 해시 → 항상 같은 파티션 → 같은 컨슈머 1개
B→A 좋아요  ─┘

컨슈머 (단일 스레드, 순차 처리):
  [1] A→B 받음 → 저장. 역방향 없음. 끝.
  [2] B→A 받음 → 저장. 역방향(A→B) 있음! → 매칭 성사 ✅

같은 쌍의 좋아요는 물리적으로 단 하나의 컨슈머 스레드에 순서대로 들어온다. B→A 처리 시점엔 A→B가 이미 완전히 커밋돼 보이는 상태 → 엇갈림이 원천적으로 불가능. 채팅의 channelId 순서보장 패턴을 도메인 정합성 문제에 재해석해 적용한 것.

Kafka를 거치면 따라오는 보너스

처리량 — 쌍이 다르면 병렬, 같은 쌍만 직렬. 초당 수만 좋아요 소화
디커플링 — 접수는 즉시 응답, 무거운 판정은 비동기. 사용자는 안 기다림
hot key 흡수 — 인기 유저 폭주를 Kafka가 버퍼(백프레셔)로 흡수
재처리/감사 — 버그 고치고 토픽 리플레이해서 매칭 재계산 가능

솔직한 트레이드오프 (면접 가산점)

  • 지연 — 매칭이 "즉시"가 아니라 비동기라 수십~수백 ms 지연 (만남앱에선 무관)
  • 최종 일관성 — 좋아요 직후 1ms 안엔 매칭 판정이 안 끝나 있을 수 있음
  • 복잡도 — Kafka/컨슈머/오프셋 관리 부담
결론: 만남 도메인은 "즉시성보다 정합성 · 확장성이 중요"하니 이 트레이드오프가 맞아떨어진다. (반대로 결제 승인 같은 동기 응답 필수 도메인이면 다른 선택을 했을 것 — "왜 이 도메인엔 이게 맞는지"까지 말하면 완벽.)

06 왜 Outbox + CDC인가 — dual-write 문제

매칭 성사 직후 해야 할 일이 둘: ① matches 저장(MySQL) ② "매칭됐어요!" 이벤트 발행(Kafka → fanout → WS 푸시 + 채팅방 오픈). 문제는 둘이 서로 다른 시스템이라는 것.

"그냥 순서대로 하면" 뭐가 문제인가

시나리오 X — DB 성공, Kafka 실패

① matches INSERT + commit → ② kafkaTemplate.send(...) 실패

매칭은 DB에 있는데 두 사람은 알림을 못 받음
→ "매칭됐다는데 채팅방이 안 열려요" 유령 매칭 ❌

시나리오 Y — Kafka 먼저, DB 나중

① Kafka 발행 → ② DB 저장 실패(롤백)

알림은 갔는데 DB엔 없음
→ "매칭 떴다가 채팅방 들어가니 없어짐" 유령 알림 ❌

두 시스템에 걸친 쓰기는 어느 순서로 해도 중간에 끊기면 어긋난다. 분산 트랜잭션(2PC)은 무겁고 느리고 Kafka 지원도 약함.

해결: Outbox + CDC

핵심 아이디어 — "Kafka에 직접 쏘지 말고, 보낼 이벤트도 그냥 DB에 같이 적어라."

1단계 — Outbox: 같은 트랜잭션 안에서 둘 다 DB에 쓰기

BEGIN TRANSACTION
  INSERT INTO matches (...)                  -- 매칭 본체
  INSERT INTO outbox (event=MATCH_CREATED,    -- 보낼 이벤트도 같은 DB에
                      recipients=[A,B], channel_id=...)
COMMIT

둘이 하나의 트랜잭션이라 같이 성공하거나 같이 실패. 어긋날 틈이 없음 (MySQL 트랜잭션 원자성).

2단계 — CDC(Debezium): 커밋된 로그를 읽어 Kafka로 배달

Debezium이 MySQL binlog(커밋 기록)를 계속 감시
  → outbox에 새 행 INSERT 감지
  → 그 행을 match-fanout 토픽으로 발행
  → fanout-delivery-service가 받아서 양쪽 WS 푸시 + 채팅방 오픈

핵심은 Debezium이 "커밋된 binlog"를 읽는다는 것. 커밋됐다 = outbox 행이 binlog에 확실히 있다 = 언젠가 반드시 Kafka로 나간다. 중간에 서버가 죽어도 재시작하면 못 보낸 지점부터 다시 읽음 → 유실이 구조적으로 불가능.

순진한 방식:  매칭저장 ──(여기서 죽으면 알림 유실)──> Kafka발행
Outbox+CDC:   [매칭저장 + 이벤트저장] 한 트랜잭션 ──binlog──> Debezium ──> Kafka발행
                  ↑ 둘이 운명공동체                       ↑ 죽어도 재개되면 반드시 배달

보너스 & 트레이드오프

  • 보너스 — 매칭 컨슈머는 DB 쓰기만 하면 됨(코드 단순·빠름). Kafka가 잠깐 죽어도 매칭 처리는 정상(이벤트는 outbox에 쌓였다 배달). 채팅 메시지에서 쓰던 패턴 그대로 재사용 → 일관된 설계
  • at-least-once — Debezium/Kafka가 같은 outbox 이벤트를 드물게 두 번 보낼 수 있음 → 소비자가 멱등해야 함. 우리는 이미 matches UNIQUE + 채팅방 멱등 생성이 있어 두 번 와도 안전 (앞에서 만든 멱등성 자산이 여기서 또 일함)
  • 비용 — 지연 · 복잡도(Kafka Connect 운영 부담)는 있지만, 유실 0 보장이 그만한 가치

07 한 줄 정리

like-relay : 동시 좋아요를 락 없이 직렬화 → 매칭 판정 (정합성)
matches+outbox : 매칭과 이벤트를 한 트랜잭션에 → 어긋남 방지 (원자성)
CDC→match-fanout: 커밋된 이벤트를 반드시 배달 → 유실 방지 (신뢰성)
세 단계가 각각 정합성 → 원자성 → 신뢰성을 책임지면서, "동시에 눌러도 / 서버가 죽어도 / Kafka가 흔들려도 매칭과 알림이 절대 어긋나거나 사라지지 않는다"를 보장한다. 이 흐름 전체를 입으로 설명할 수 있으면 분산 시스템 면접 한 챕터는 끝.
